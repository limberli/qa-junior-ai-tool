# QA Junior AI Tool

Мультиагентная система для анализа документов с требованиями к программному обеспечению. Генерирует структурированные тест-кейсы и анализ рисков с помощью локальных LLM через Ollama.

## Что делает проект

Отправьте документ с требованиями (пользовательскую историю, спецификацию фичи) на REST API. Система параллельно передаёт документ двум AI-агентам:

- **Tester Agent** — генерирует структурированный план тестирования: позитивные/негативные/граничные/безопасностные/нагрузочные тест-кейсы в формате `TC-NNN | Название | Предусловия | Шаги | Ожидаемый результат | Приоритет`
- **Analyst Agent** — производит анализ рисков: выявляет неоднозначности, недостающие требования и потенциальные точки отказа

Оба ответа агрегируются, сохраняются в БД и возвращаются вызывающей стороне.

### Как сервисы общаются

Все вызовы агентов используют протокол **A2A (Agent-to-Agent)** — открытый стандарт взаимодействия агентов на основе **JSON-RPC 2.0**. Каждый агент предоставляет:

- `POST /` — метод `tasks/send`: принимает документ, возвращает завершённую задачу с текстовыми артефактами
- `GET /.well-known/agent-card.json` — метаданные агента (название, возможности, навыки)

Host Orchestrator при запуске обнаруживает агентов через их agent card, затем вызывает обоих агентов **параллельно** с помощью `CompletableFuture`. Каждый вызов защищён цепочкой Resilience4j: **Circuit Breaker → Time Limiter (30 с) → Retry (3 попытки, экспоненциальная задержка)**.

## Стек технологий

| Слой | Технология |
|---|---|
| Backend-фреймворк | Spring Boot 3.3.5 |
| Интеграция с LLM | Spring AI 1.0.0-M6 (OpenAI-совместимый провайдер) |
| LLM-рантайм | Ollama (`phi3:mini`) или Groq cloud API (`llama-3.1-8b-instant`) |
| Протокол агентов | A2A Protocol (JSON-RPC 2.0) |
| Отказоустойчивость | Resilience4j (circuit breaker, retry, time limiter) |
| Хранилище | JPA + H2 (dev) / PostgreSQL (prod) |
| Документация API | springdoc-openapi (Swagger UI по адресу `/swagger-ui`) |
| Фронтенд | Next.js 15, MUI 7, TypeScript |
| Сборка | Maven (мультимодульный), Docker |

## Запуск проекта

### Предварительные требования

- Docker и Docker Compose
- ~6 ГБ свободного места на диске (для модели `qwen2.5:7b`)

### Быстрый запуск (скрипты)

Проще всего — через лаунчер. Он спросит язык (RU/EN), режим (Groq / локальная Ollama),
при необходимости попросит ввести `GROQ_API_KEY`, поднимет стек, дождётся готовности и
откроет браузер.

```bash
# macOS / Linux
./run.sh          # запуск (интерактивно)
./stop.sh         # остановка
```

```bat
REM Windows — двойной клик по run.bat, либо из консоли:
run.bat
stop.bat
```

Дальше — то же самое вручную через Docker Compose:

### Вариант 1 — Полный стек с проверками состояния (рекомендуется)

```bash
docker compose up --build
```

Сервисы запускаются в порядке зависимостей. При первом запуске Ollama загружает модель `qwen2.5:7b` (~4 ГБ) — это занимает несколько минут. Все сервисы имеют healthcheck, поэтому оркестратор ждёт готовности агентов.

### Вариант 2 — Лёгкий запуск (phi3:mini, Ollama локально)

```bash
docker compose -f docker-compose-simple.yml up --build
```

Использует модель `phi3:mini` вместо `qwen2.5:7b`. Без healthcheck — после запуска подождите немного перед отправкой запросов.

> **Внимание:** на CPU без GPU модель работает медленно (2–10 минут на запрос). Если нужна скорость — используйте Вариант 3.

### Вариант 3 — Groq API (быстро, бесплатно, без GPU)

Groq предоставляет бесплатный доступ к облачным LLM с очень быстрым инферансом. Ollama при этом не нужна.

**Шаг 1.** Зарегистрируйтесь на [console.groq.com](https://console.groq.com) → API Keys → Create API Key.

**Шаг 2.** Положите ключ в файл `.env` в корне репозитория (он уже в `.gitignore`). Docker Compose читает `.env` автоматически:

```bash
echo "GROQ_API_KEY=gsk_...ваш_ключ..." > .env
```

**Шаг 3.** Соберите образы и запустите стек:

```bash
# Первый запуск (со сборкой)
docker compose -f docker-compose-groq.yml up --build -d

# Последующие запуски (образы уже собраны)
docker compose -f docker-compose-groq.yml up -d --no-build
```

По умолчанию используется модель `llama-3.1-8b-instant`. Время ответа — 1–3 секунды.
Веб-интерфейс на `http://localhost:3000` автоматически подключает оркестратор при старте
(через `NEXT_PUBLIC_DEFAULT_AGENT_CARDS_URL`, указывающий на `/agent-cards.json`).

> **Примечание:** в Groq-режиме `/actuator/health` оркестратора показывает `DOWN`, потому что
> его health-check Ollama не может достучаться до локальной Ollama. Это косметика — генерация
> работает нормально (оба агента `UP`).

**Как вернуться на локальную Ollama:**

```bash
# Остановить Groq-стек
docker compose -f docker-compose-groq.yml down

# Запустить Ollama-стек
docker compose -f docker-compose-simple.yml up --build
```

Данные истории анализов (`orchestrator_data`) сохраняются между перезапусками — переключение между режимами не удаляет историю.

### Доступные эндпоинты после запуска

| Сервис | URL |
|---|---|
| Отправить документ на анализ | `POST http://localhost:8080/api/analyze` |
| История анализов (с пагинацией) | `GET http://localhost:8080/api/history` |
| Карточки обнаруженных агентов | `GET http://localhost:8080/api/agents` |
| Схема настроек генерации (для UI) | `GET http://localhost:8080/api/modes` |
| Swagger UI | `http://localhost:8080/swagger-ui` |
| Веб-интерфейс | `http://localhost:3000` |

### Подключение веб-интерфейса к оркестратору

1. Открыть **http://localhost:3000**
2. Нажать кнопку **"Add Agent"** в шапке
3. Ввести URL агента и нажать **Add Agent**:
   ```
   http://host-orchestrator:8080/.well-known/agent-card.json
   ```
4. В списке появится **"QA Analysis Orchestrator"** — выбрать его и отправить текст требований

> **Почему `host-orchestrator:8080`, а не `localhost:8080`?**
> Все сервисы работают внутри Docker. Прокси-запросы от a2a-ui выполняются серверной стороной Next.js *внутри контейнера*, где `localhost` — это сам контейнер a2a-ui, а не оркестратор. Имя `host-orchestrator` — это имя сервиса в Docker-сети, по которому контейнеры находят друг друга.

```bash
curl -X POST http://localhost:8080/api/analyze \
  -H "Content-Type: application/json" \
  -d '{"documentText": "Как пользователь я хочу входить в систему по email и паролю, чтобы получить доступ к своему аккаунту."}'
```


## Предварительные настройки генерации тест-кейсов

Перед отправкой требования веб-интерфейс показывает **экран настроек**, на котором можно
задать, как именно генерировать тест-кейсы. Настройки **не вклеиваются в текст** на стороне
фронта — они едут как **структурированные A2A-метаданные** (`message.metadata.qa`) и
детерминированно собирают system-prompt агента на бэке. За счёт этого инструмент именно
конфигурируемый, а не тонкая обёртка над LLM.

### Контролы (режим `test-cases`)

| Настройка | Тип | Опции | По умолчанию |
|---|---|---|---|
| Тип тест-кейса (`caseTypes`) | мульти | UI, API, БД, Комбинированный | UI |
| Техника тест-дизайна (`techniques`) | мульти | Эквивалентное разбиение, Граничные значения, Таблица решений, Pairwise, Позитив/негатив | Граничные значения + Позитив/негатив |
| Детализация шагов (`stepDetail`) | один из | short (3–5), medium (6–10), detailed (10+) | medium |

### Как это устроено

```
Экран настроек (UI)
  → message.metadata = { qa: { mode, selections } }      (A2A message/send)
  → оркестратор читает params.metadata и прокидывает в tester-agent
  → QaPromptBuilder собирает system-prompt из qa-prompts.yml + выбранных опций
  → Ollama/Groq → markdown-таблица: | ID | Описание проверки | Шаг | Ожидаемый результат |
```

Пример payload `message/send` (`POST /`):

```jsonc
{
  "jsonrpc": "2.0", "id": "ui-1", "method": "message/send",
  "params": {
    "message": {
      "role": "user", "kind": "message", "messageId": "m1", "contextId": "c1",
      "parts": [{ "kind": "text", "text": "<текст требования>" }]
    },
    "metadata": {
      "qa": {
        "mode": "test-cases",
        "selections": {
          "caseTypes": ["api"],
          "techniques": ["boundary", "pairwise"],
          "stepDetail": ["short"]
        }
      }
    }
  }
}
```

### Единый источник правды: `qa-prompts.yml`

И схема контролов для UI (отдаётся через `GET /api/modes`), и фрагменты system-prompt лежат
в **`tester-agent/src/main/resources/qa-prompts.yml`**. У каждой опции есть поле `prompt:`;
`QaPromptBuilder` склеивает `base` → фрагменты выбранных опций → `output`. Если какой-то
контрол не передан — берутся `defaults` из каталога, поэтому промпт всегда полный.

**Добавить новую технику/тип:** добавьте один `option` (с его `prompt`) под нужным контролом
в `qa-prompts.yml`. Он автоматически появится в UI и начнёт влиять на промпт — без правок
Java или фронтенда. Новые режимы (тест-план, чек-лист, кодер) добавляются так же — как записи
под `qa.modes`.

Фронтенд рендерит экран настроек динамически по `GET /api/modes`, а при недоступности бэка
откатывается на встроенный конфиг (`a2a-ui/src/config/modes.ts`).

## Локальная разработка (без Docker)

Запустите Ollama локально, затем поднимите каждый модуль:

```bash
# Терминал 1
cd tester-agent && mvn spring-boot:run

# Терминал 2
cd analyst-agent && mvn spring-boot:run

# Терминал 3
cd host-orchestrator && mvn spring-boot:run -Dspring-boot.run.profiles=dev
```

Профиль `dev` использует H2 in-memory БД — внешняя база данных не нужна.

### Запуск тестов

```bash
# Интеграционные тесты (агенты заменены WireMock-заглушками — Ollama не нужен)
mvn test -pl host-orchestrator
```

### Фронтенд (a2a-ui)

```bash
cd a2a-ui
npm install
npm run dev   # http://localhost:3000
```
